项目名称 Math Quiz Multi-Agent（数学试题多智能体仿写系统） 文档版本 v1.0（2026-07-07） 读者对象 项目组成员、研学指导老师；同时作为后续开发（Codex 执行）的依据 上游文档 01 号产品规格介绍书（定义"做什么"，本文档回答"怎么做"）

一、设计目标与约束

1.1 已有资产

项目已有一版可运行的命令行程序，位于 /Users/jingj/Downloads/math-quiz-multi-agent/（原始开发环境为 Windows），包含：

  • 四个智能体程序（decomposer / reviewer / generator / qc 各一个 Python 文件，合计约 1100 行）；
  • 总控 run_pipeline.py，实现了两道"审核不合格就打回"的闭环，每道闭环最多打回 2 次；
  • common.py 公共库，实现了 .env 读取、DeepSeek API 调用、调试输出落盘，以及非法 JSON 的自动修复（历史上 Reviewer 曾输出内嵌未转义引号的坏 JSON 导致整条流水线崩溃，这个修复补丁是现版本唯一打过的补丁）；
  • 一次完整的成功运行记录（outputs 目录下的 round 与 final 文件，QC 评分 95 分通过）。

设计原则是在这套已验证可行的骨架上做加法，四个智能体的职责边界和文件接口保持不变。

1.2 硬约束

约束 内容 由来
零第三方依赖 只用 Python 标准库，不 pip install 任何包 开发机 pip 走代理报错，装包不可靠
运行平台 Windows 笔记本（i5-8265U / 24GB 内存 / 集成显卡） 用户现有设备
大模型服务 DeepSeek API（文本），另有一个已开通的视觉大模型 API（拍照识别用） 已有账号
本地不跑模型 一切模型推理都在云端 集成显卡与低压 CPU 不具备本地推理条件

1.3 设备条件对方案的影响

i5-8265U 是 15 瓦低压四核处理器，128MB 显存说明只有集成显卡。这个配置排除了两类方案：本地部署任何开源模型（含本地 OCR 大模型），以及任何依赖 GPU 的图像处理。但它完全胜任另一类工作：发 HTTP 请求、读写文件、让浏览器渲染网页。所以总体思路定为本机只做调度与展示，计算全部外包给云端 API。这既是无奈，也刚好是多智能体架构的自然形态。

数据一律落盘为文件（JSON 与 Markdown），沿用现有 outputs 目录约定，每一轮打回都留档。这让任何一次运行都可复盘，也让测试可以直接对文件断言。

三、关键选型与理由

3.1 界面路线（决策记录）

三个候选的对比：

候选 优点 缺点 结论
A. 标准库 Web 服务 + 单页 HTML + 本地 KaTeX 零依赖，界面自由度最高，效果上限高 需要自己写路由和前端 采用
B. Streamlit 写得快 必须过 pip 这一关；页面是模板脸，观感撞车 放弃
C. 只输出静态 HTML 报告 最省事 没有交互，拍照上传、专家模式都无处安放 作为 F10 导出功能保留

选 A 的核心理由有三个。其一，pip 在开发机上不可靠是既成事实，方案 A 是唯一完全绕开它的路线。其二，产品要求"得体、有质感"的界面，自己写的 HTML/CSS 可以完全控制观感，Streamlit 反而受限。其三，这台机器跑浏览器绰绰有余，KaTeX 渲染发生在浏览器里，服务端毫无压力。

KaTeX 本身不装包：官网发布的是一组静态文件（js、css、字体，合计约 3MB），下载后放进项目 static/ 目录，网页本地引用即可，离线可用。

3.2 模型分工（成本与准确率的平衡）

现版本所有智能体都用 deepseek-chat。第一阶段改为按任务性质分配两种模型：

智能体 任务性质 模型 理由
Decomposer 阅读理解、信息抽取 deepseek-chat 不需要深度推理，chat 便宜且快
Reviewer 核对分析是否准确 deepseek-chat 同上
Generator 出题并解题 deepseek-reasoner 解题是纯数学推理，reasoner 数学能力显著更强
QC 独立重解并验答案 deepseek-reasoner 验算和解题同性质，弱模型验不出强错误

这是针对"Token 花了答案还不对"这一核心困境的第一刀：把贵而强的推理模型只用在真正需要推理的两个环节。deepseek-reasoner 与 deepseek-chat 同属一个 API，只改请求里的模型名即可切换，注意 reasoner 不支持 response_format 强制 JSON 参数，需要靠提示词约束加上现有的 JSON 修复机制兜底。模型名写进 .env 配置而非写死在代码里，方便日后换更强的模型做对照实验。

程序侧再加两道不花 Token 的把关（详见 4.4）：题号数量核对、schema 校验。能用程序判断对错的事，都从大模型手里拿走。

3.3 拍照录入（视觉输入适配层）

新增 vision_input.py，职责单一：把一张照片变成一段可进流水线的题目文本。

  • 接口约定。输入为图片文件；输出为固定结构的 JSON：
{
  "text_markdown": "识别出的题目全文，数学公式用 LaTeX 表示",
  "has_figure": false,
  "confidence": 0.95,
  "warnings": ["图片略有倾斜，已按内容纠正阅读顺序"]
}
  • 实现方式。视觉大模型 API 走 OpenAI 兼容格式：图片读入后 base64 编码，放进消息体一起 POST，标准库 urllib 即可完成，与 common.py 现有调用方式同构。API 地址、密钥、模型名全部放 .env（VISION_API_BASE / VISION_API_KEY / VISION_MODEL），换供应商不改代码。
  • 分流规则。has_figure 为真（题目依赖几何图或坐标系图）时，界面提示"当前版本暂不支持带图题目"，任务终止；confidence 低于阈值时提示用户重拍。这两条边界行为都有对应测试用例。

3.4 公式渲染

  • 选 KaTeX 而非 MathJax：两者都能把 LaTeX 渲染成印刷体公式，KaTeX 体积更小、渲染更快，覆盖的语法范围对中学数学绰绰有余。
  • 渲染发生在三处：原题识别结果、解答步骤、新题与答案。三处共用同一个渲染函数。
  • 大模型输出的 LaTeX 偶有小错（如括号不配对），渲染层开启 KaTeX 的容错模式，渲染失败的公式降级显示原文并标黄提醒，而非整页报错。

四、模块设计

4.1 保持不变的部分

四个智能体的系统提示词结构、输入输出 JSON schema、outputs 目录的 round/final 文件命名，全部沿用现状。run_pipeline.py 的双闭环判定逻辑（合格分数阈值 90、最多打回 2 次）也不动，只做两处扩展：接受"从内存对象而非固定路径读输入"的参数化改造，以及在闭环节点上报进度事件。

4.2 新增模块

server.py（服务层）。 标准库 http.server + threading。路由表：

方法与路径 作用
GET / 返回单页应用 index.html
GET /static/... KaTeX 等静态资源
POST /api/job 提交任务（照片 base64 或题目文本），返回 job_id
GET /api/job/{id}/status 查询进度（前端每 2 秒轮询一次）
POST /api/job/{id}/confirm 专家模式下提交人工修改后的分析，流水线继续
GET /api/job/{id}/result 取最终结果

任务状态机：queued → vision → decompose → review → (await_confirm) → generate → qc → done / failed / rejected_figure。每个任务在 outputs/jobs/{job_id}/ 下有独立目录，流水线在后台线程中执行，HTTP 层永不阻塞。

validators.py（schema 校验）。 手写校验函数（零依赖），对四个智能体的输出逐字段查类型、查取值范围（如 score 必须是 0 到 100 的整数）。校验失败视同"输出不合格"，走既有的打回或修复路径。这补上了现状里"全靠提示词约束"的缺口。

budget.py（Token 台账）。 DeepSeek 每次响应都带用量字段，台账逐次累加、按智能体分项记录，写入任务目录。超过单任务限额（默认 20 万 Token，可配）时中止流水线，保留已完成的 round 文件并明确报错。界面上的 Token 计量条也由它供数。

render_report.py（报告导出）。 把最终结果拼装成自包含 HTML（内嵌 KaTeX 资源），对应功能 F10，可打印、可留存、可脱网打开。

4.3 专家模式

产品规格里的 F9，回应"有些信息想手动调，但用户端做不到"的历史问题。设计为一个提交任务时的开关：

  • 关（默认）：流水线一跑到底，全自动。
  • 开：闭环一定稿后，状态停在 await_confirm，界面把考点、难度、题型、出题约束展示为可编辑表单；用户改完提交，流水线以修改后的分析为准继续走闭环二。改动会记录进任务目录，供复盘对比"人工修正了什么"。

这个设计顺带成为答辩时的互动亮点：现场请观众改一个考点，看新题跟着变。

4.4 稳定性设计

按"错误要能定位、流程要能兜底"两条要求组织：

  1. JSON 修复。沿用现有机制（原始输出先落盘 debug、loads 失败自动调用修复提示词重试），所有调用都带 agent_name，坏输出能定位到具体智能体。
  2. schema 校验。见 validators.py，防"合法 JSON 但字段错"这类修复机制管不到的问题。
  3. 程序规则把关。题目数量与答案数量必须相等、题号必须一一对应、LaTeX 定界符必须配对，这三条用纯程序检查，不合格直接打回 Generator，不消耗 QC 的 Token。
  4. 轮次与限额双保险。打回次数封顶（现有）+ Token 限额（新增），保证流水线在任何情况下都会停。
  5. 全程落盘。每轮中间产物、每次原始输出、每笔 Token 用量都有文件，出问题看 outputs/ 而非猜。
  6. Windows 编码防线。所有文件读写显式 encoding="utf-8"（现有代码已做到），服务端响应头显式 UTF-8，控制台输出避免依赖系统默认编码（Windows 中文环境默认 GBK，是历史上大量"乱码玄学"的来源）。

五、界面设计

观感目标一句话：像一份印刷得讲究的数学讲义，配上克制的现代工程感。得体，不花哨。

布局。 单页三区：

┌────────────────────────────────────────────────────┐
│  顶栏  项目名 · 专家模式开关 · Token 计量条           │
├──────────────┬─────────────────────────────────────┤
│  输入区       │   流水线时间线                        │
│  拍照/选图    │   ○ 识别 → ○ 分析 → ○ 审核           │
│  或粘贴文本   │   → ○ 出题 → ○ 质检                  │
│  [开始]      │   （当前环节高亮，打回时显示轮次徽标）  │
├──────────────┴─────────────────────────────────────┤
│  结果区（标签页）                                     │
│  原题识别 │ 考点分析 │ 解答 │ 新题 │ 新题答案 │ QC报告 │
└────────────────────────────────────────────────────┘

视觉方向。 浅色纸感底、深藏蓝主色、琥珀色点缀作强调；题目与答案用卡片承载，投影环境下保持高对比；中文用系统自带的苹方/微软雅黑字体栈，公式一律 KaTeX，杜绝截图贴公式。具体色值和间距在实现任务书中给出，允许实现时微调。

关键交互。 长耗时环节（reasoner 解题可达数分钟）必须有进度反馈：时间线节点做呼吸动画、显示已用时与已耗 Token。打回发生时，时间线上出现"第 2 轮"徽标并可点开查看打回原因。这些细节是"质感"的主要来源，也是向评委展示闭环机制的可视化窗口。

六、成本与耗时估算

单题全流程的量级测算（以现有运行记录外推，含一次打回的余量）：

环节 模型 估算 Token（输入+输出）
视觉识别 视觉大模型 2 千左右
分析 + 审核（闭环一，含打回） chat 1 万左右
出题解题 + 质检（闭环二，含打回） reasoner（含思维链） 3 万至 6 万
合计 4 万至 8 万

按 DeepSeek 现行牌价折算，单题成本在几分到几毛钱人民币的量级（reasoner 输出计费高于 chat，是大头）。精确单价随官方调价浮动，此处不写死数字，实施时以官网牌价核算，这项核实工作已列入 GPT 的调研任务（见 03 号文档 T07）。一次课堂演示按 10 道题计，预算控制在几元以内，可接受。

耗时方面：chat 环节每次调用数秒到半分钟；reasoner 解一道中学难题约 1 到 5 分钟。全流程 2 到 10 分钟，这决定了界面必须异步加进度展示（见第五节），也决定了答辩演示要提前跑一道"热场题"备着。
