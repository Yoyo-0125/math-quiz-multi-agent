一、这个产品是什么

一句话说清：拍一张数学题的照片，系统自动读题、解题、分析考点，再仿写出一道考点相同、难度相当的新题，并附完整答案。全过程由四个互相监督的 AI 智能体协作完成。

设想这样一个场景。班里每位同学自备一份数学试卷，从中挑一道题拍照上传。几分钟后，屏幕上依次出现：

  1. 原题的识别结果，公式排版工整，和课本印刷体一样；
  2. 这道题考什么知识点、大致什么难度；
  3. 一份带步骤的详细解答；
  4. 一道"孪生新题"，考点相同、难度相当，但数字和情境都换过，附独立的答案页。

这就是本系统第一阶段要完成的全部能力。

二、要解决的问题

这个产品回应三个真实的痛点。

找同型题难。 一道题做错了，想再练三道同款巩固，只能翻题库、翻教辅，效率很低，还不一定找得到难度合适的。

出变式题费时。 老师想给某个学生出针对性的变式练习，需要自己改编题目、重新演算答案，一道题往往要花十几分钟。

单个 AI 直接出题不可靠。 直接让一个大模型出题，题目和答案经常有错，而且错了没人发现。我们在实验中确认过这一点：模型给出的答案看起来头头是道，实际算错的情况并不少见。

本系统的对策是"分工加互查"。把出题任务拆给四个各司其职的智能体，分析结果有人审，生成结果有人验，不合格就打回重做，再配上程序规则做最后兜底。单个 AI 会犯的错，在流水线里有多次被拦下的机会。

三、用户与使用场景

用户 场景 使用方式

四、系统如何工作

系统的核心是四个智能体（Agent，即分工明确的 AI 程序单元）组成的流水线，外加两道"打回重做"的闭环：

试卷照片 ──▶ 视觉识别 ──▶ 题目文本（Markdown + LaTeX 公式）
                              │
                              ▼
               ┌── 闭环一 ──────────────────┐
               │  Decomposer  读题、分析考点 │
               │        │                   │
               │        ▼                   │
               │  Reviewer   审核分析质量   │──不合格──▶ 打回重做（最多 2 次）
               └────────┬───────────────────┘
                        ▼  分析定稿（专家模式下可人工确认、修改）
               ┌── 闭环二 ──────────────────┐
               │  Generator  出新题并解答   │
               │        │                   │
               │        ▼                   │
               │  QC        验题、验答案    │──不合格──▶ 打回重做（最多 2 次）
               └────────┬───────────────────┘
                        ▼
        新题 + 答案（含步骤）+ 考点报告 ──▶ 网页渲染展示

"打回"的意思是：审核方给出具体问题清单，流水线把清单连同原始材料一起交还给上一个智能体重做。审核方只提意见，从头到尾都由生成方自己改，两边职责分开。这个机制正是本项目区别于"直接问一次 AI"的地方。

六、明确不做什么

边界写清楚，和做什么同样重要：

  • 依赖图形的题目。第一阶段检测到题目含"如图"或识别出图形时，界面明确提示"当前版本暂不支持带图题目"，而非硬着头皮算错。
  • 只处理中学数学，物理、化学等其他学科不做。
  • 单机本地运行，不做云端部署、不做多人账号。
  • 以单道题为处理单位，整卷批量处理不做。

七、运行环境与非功能要求

硬件。 用户现有笔记本电脑（Intel i5-8265U、24GB 内存、集成显卡）即可运行。所有重计算都发生在云端大模型一侧，本机只负责发请求、收结果和网页显示，对配置几乎没有要求。

软件。 Windows 系统，Python 标准库，零第三方安装（此约束的由来见设计方案第一节），另需一个现代浏览器（Edge 或 Chrome）。

网络。 运行时需联网调用大模型 API（应用程序接口，即按次付费的云端 AI 服务）。

成本。 单道题全流程（含打回重做）预计消耗数万 Token，按 DeepSeek 现行牌价折合几分到几毛钱人民币。此为量级估算，精确单价以官网当日价格为准，详细测算见设计方案第六节。

速度。 推理型模型解难题较慢，单题全流程预计 2 到 10 分钟。界面全程显示进度，不会出现"点了没反应"的白屏等待。

安全。 API 密钥保存在本机 .env 文件中，不写进代码、不上传代码仓库、不出现在界面和日志里。

八、验收标准

第一阶段做完的判定条件，就是现场能完整走通下面这份演示清单：

  1. 拍摄一道打印体代数题，系统正确识别为带公式文本；
  2. 屏幕展示该题的考点分析；
  3. 展示含步骤的解答；
  4. 生成新题与答案，并通过 QC 审核；
  5. 全程公式渲染达到印刷标准，界面观感得体；
  6. 历史上曾令系统崩溃的"含参分式不等式"作为回归用例正常通过；
  7. 测试集中标注 P0 的用例全部通过（用例清单见 04 号文档）。
