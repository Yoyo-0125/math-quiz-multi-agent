二、失败样例（历史故障与由此而来的两条设计）

真实故障记录：对复杂题（含参分式不等式）运行时，Reviewer 在输出 JSON 的字符串里直接嵌了未转义的英文双引号：

"minor_issues": [
    "题型"解不等式"过于宽泛，应更具体为"含参数分式不等式求解"",
    ...
]
json.decoder.JSONDecodeError: Expecting ',' delimiter: line 6 column 9 (char 87)
RuntimeError: 模型没有返回合法 JSON

整条流水线在此崩溃。排查依据 debug 落盘文件定位到故障节点是 Reviewer 的输出解析（并非 API 请求失败，也非 Decomposer 写文件失败）。由此产生两条至今有效的设计：

  1. 提示词约束。所有智能体的提示词禁止在字符串里使用英文双引号，引用概念改用中文引号或书名号；
  2. 程序兜底。common.py 在 JSON 解析失败时自动落盘原始输出并调用修复流程重试，多次失败才报错。

这个案例说明"大模型的输出必须当作不可信输入处理"，也是测试集里 T-JSON-01 和 T-INP-03 两个用例的由来。

三、测试集设计原则

  1. 每个接口至少一条用例。 接口清单：视觉输入适配、Decomposer 首轮与重做、Reviewer、Generator 首轮与重做、QC、闭环控制（轮次上限、Token 限额）、JSON 修复、schema 校验、程序把关规则、渲染、Web 服务与专家模式、密钥安全。
  2. 历史故障必须变成回归用例。 曾经翻过车的含参分式不等式永久保留在测试集里，每个里程碑跑一遍。
  3. 判定方式优先程序化。 能写断言的（文件存在、字段类型、数量相等）用程序判；只能人看的（答案对不对、公式漂不漂亮）明确写"人工"并给核对要点。
  4. 打回路径要能人为触发。 用"故意做坏的输入"测试打回机制，而非祈祷模型恰好出错。
